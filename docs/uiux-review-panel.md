## 第1部: ファーストインプレッション（5分操作後の所感）

### 田中（D）— デザイナー

> 全体のトーン（モノクロ＋ミニマル）は統一されていて好感触。ただし「統一されすぎて」情報の優先度が崩壊している。すべてが同じ温度感で並んでいるため、どこを見ればいいか分からない。Tonal のリファレンスを参考にしている割に、Tonal の肝である「コントラストの強弱による誘導」が再現できていない。

**具体的な問題点:**
- ダッシュボードの StatCard 3つが `grid-cols-3` で等幅に並んでいるが、「今週のセッション」と「最新の体重」の重要度は明らかに異なる。等しいサイズは情報設計の放棄。
- `text-[22px]` のタイトルと `text-[11px]` のラベルの組み合わせが全ページで使い回されているが、機械的すぎて「設計した」感がない。
- カードの `shadow-[0_0_0_1px_rgba(0,0,0,.04)]` は極めて薄く、Surface色 `#f3f3f3` との差が 実質3%未満。目を凝らさないとカードの境界がわからない。
- FAB が `h-12 w-12` で iPhone SE では指で隠れるサイズ。iOS HIG 推奨の最低タップ領域 44pt をギリギリ満たすが、FAB としては小さすぎる。

### 佐藤（UA）— 筋トレ歴2年

> Strong を普段使ってるけど、このアプリは「何ができるのか」が最初の画面から全然分からない。ダッシュボードを開いても「今日もいこう」って言われるだけで、今日やるべきワークアウトが出てこない。「動画ライブラリ」のカードがあるけど、俺は筋トレの記録がしたいだけなのに動画推しなのかな？と混乱した。

**具体的な問題点:**
- ダッシュボードに「今日のワークアウト」概念がない。ルーティン管理がゼロ。毎回ゼロからワークアウトを組むのか？
- 「すべて見る →」を押すとワークアウト履歴に飛ぶが、そこから記録開始までが「FABタップ → ブロック作成 → 種目追加 → 設定 → 追加 → 保存」で最低6タップ。Strong は2タップで記録開始できる。
- ワークアウト編集画面のタイトル「マイワークアウト」がデフォルトのまま。毎回変える人がどれだけいるか。日付+時間のオート命名が欲しい。

### 鈴木（UB）— 初心者

> 正直、最初に見た感想は「おしゃれだけど何をすればいいかわからない」。ワークアウトのタブを押すと履歴が出るけど、初めて使う人には履歴がないから真っ白画面になる。「鉛筆マークから作成しよう」って書いてあるけど、その鉛筆マークがどこにあるか一瞬見つからなかった。右下にあるのか…。

**具体的な問題点:**
- 空状態（Empty State）が弱い。アイコンとテキストだけ。オンボーディングやチュートリアルが一切ない。
- 種目一覧で「ベンチプレス」「インクラインプレス」と出てくるが、初心者は何がどう違うかわからない。画像やイラストが一切ない。`descJa` があるが1行の文章だけで、フォーム指導にはならない。
- 「ブロック」という概念が意味不明。「ブロック1」と表示されても、それが何を意味するのか説明がない。
- ウォームアップのトグルの説明「最初の1セットを軽めの負荷で行います」は、初心者にはウォームアップの重量をどう設定すればいいのか分からない。

### 高橋（UC）— ガジェット好きエンジニア

> 技術的には Next.js App Router + Tailwind で手堅いけど、実装の都合がUIに漏れている箇所が多い。`sessionStorage` でビデオURLを渡してるのは分かるけど、ブラウザリロードで消える。PWA で使うこと前提なら致命的。あと全データが mock で、永続化レイヤーがゼロ。この状態で「ローンチ」は語れない。

**具体的な問題点:**
- データ永続化が一切ない。`console.log` で出力して終わり。ローカルストレージすら使っていない。
- 撮影画面が `calc(100dvh - 56px)` + 負の margin で BottomNav の下に潜り込む hack。脆弱すぎる。
- BottomNav の z-index が `z-50`、Body シートが `z-[110]` — マジックナンバー。スケールしない。
- `formatRecordDate` がハードコードで `TODAY = "2026-04-21"` なのに、一部は `new Date()` を使っている。テスト不可能な時限爆弾。

### 山田（PM）

> プロダクトとして見ると、MVPの「最低限何ができるか」が不明確。記録しても保存されない。動画を撮っても消える。これはデモであってMVPですらない。ユーザーリテンションの観点で言うと、DAU が落ちる3大要因「初日体験の弱さ」「記録の煩雑さ」「成果の見えなさ」をすべて抱えている。

**具体的な問題点:**
- **初日体験**: オンボーディングゼロ。ユーザーは自力で「タブを押す→FABを見つける→ブロックを理解する→種目を探す」という探索をしなければならない。
- **記録の煩雑さ**: 1ワークアウト記録に必要なタップ数が多すぎる。競合比較で Strong は「テンプレート選択→開始→完了」の3ステップ。
- **成果の見えなさ**: ダッシュボードに「前回比」「ベスト記録」「週間ボリューム推移」がない。ユーザーは自分の成長を確認できない。
- **リテンション施策ゼロ**: プッシュ通知、リマインダー、ストリーク、目標設定、いずれも未実装。
- **マネタイズ導線なし**: 仮にローンチしても収益化の余地がどこにもない。

### 中村（TA）— トレーナー8年

> クライアントに使わせることを想定すると、致命的に足りないのは「テンプレート/ルーティン」機能。トレーナーがプログラムを組んで渡して、クライアントがその通りにやる、という最も基本的なユースケースが成立しない。毎回ゼロからワークアウトを組むのは非現実的。

**具体的な問題点:**
- テンプレート保存機能がない。「プッシュデー」「プルデー」のような定型メニューを再利用できない。
- RPE（主観的運動強度）の入力フィールドがワークアウト編集画面にない。Capture meta にだけ存在するのは設計ミス。
- スーパーセット/ドロップセット/ジャイアントセットの概念がない。「ブロック」がそれを担うのだとしたら、ブロック内に複数種目を入れたときの実行順序が不明。
- レスト（休憩時間）タイマーがない。記録アプリとして致命的。

### 木村（TB）— フィジーク選手

> データ重視の立場から言うと、記録の粒度が粗い。セットごとの重量×レップスは入力できるけど、セット間のレスト時間、テンポ（挙上速度）、1RM推定値、部位別ボリューム集計、いずれも欠けている。BodyDetail のチャートは体重だけで、筋囲や写真比較がない。フィジーク勢が使うレベルではない。

**具体的な問題点:**
- 1RM 自動計算がない（Epley式/Brzycki式）。
- ボリューム（重量×レップス×セット）のグラフが「総重量」として HistoryCard に出るが、部位別・種目別の推移が見れない。
- 体組成として体重と体脂肪率しか記録できない。筋肉量、骨格筋率、各部位の筋囲が入力できない。
- 写真で経過を記録するフォトログ機能がない。フィジーク選手にとっては数値より視覚的変化が重要。

### 伊藤（TC）— 初心者指導メイン

> 初心者目線で言うと、このアプリは「上級者が作った上級者のためのアプリ」になりかけている。種目名がカタカナ＋1行説明では、初心者は「ラットプルダウン」と「シーテッドロウ」の違いが分からない。最低でも種目ごとに対象筋群のイラストか、短いデモ動画のサムネイルが欲しい。

**具体的な問題点:**
- 種目選択画面に画像が一切ない。テキストオンリーのリストは初心者の離脱ポイント。
- カテゴリが「胸/背中/肩/腕/脚」の5分類だが、「体幹（コア）」「臀部」がない。初心者向けの種目（プランク、ヒップスラストなど）がモックデータにも存在しない。
- マシンかフリーウェイトかの区別がない。初心者はマシンから始めることが多いのに、フィルタリングできない。
- フォーム確認動画との連携が弱い。撮影した動画を後から特定の種目・セットに紐付ける導線がない。

## 第2部: 画面別の詳細レビュー

### 2-1. ダッシュボード (`/dashboard`)

| 問題 | 深刻度 | カテゴリ |
|------|--------|----------|
| StatCard が3等分で情報優先度が不在 | 中 | 情報設計 |
| 「今日もいこう」のコピーが曖昧。時間帯による挨拶分岐もない | 低 | マイクロコピー |
| 「最近のワークアウト」が1件しか表示されない | 中 | 情報量 |
| 設定アイコンが右上にあるが、視認性が低い（テキストラベルなし） | 低 | ナビゲーション |
| ユーザーの名前が一切表示されない（パーソナライゼーション欠如） | 中 | エンゲージメント |
| 「動画」セクションがダッシュボードに必要か疑問。BottomNav にない導線の補完としては弱い | 中 | 情報設計 |
| 今日のスケジュール/次のワークアウト提案がない | 高 | コア機能欠如 |

**田中（D）の提案:**
> ダッシュボードは「今日あなたがやるべきこと」にフォーカスすべき。StatCard を廃止して、代わりに「今日のワークアウト」カード（テンプレートから自動提案）をメインに据える。統計はセカンダリとして下部にまとめる。

### 2-2. ワークアウト履歴 (`/workouts`)

| 問題 | 深刻度 | カテゴリ |
|------|--------|----------|
| 履歴カードがすべて同じ見た目。直近と1週間前の区別がつかない | 中 | 視認性 |
| カードをタップしても何も起きない（詳細ページがない） | 高 | 機能欠如 |
| 検索・フィルター機能がない（日付、カテゴリ、種目で絞れない） | 高 | ユーザビリティ |
| 「全 5 件」のカウントがヘッダー右端にあるが、情報として弱い | 低 | UI |
| 各カードの Metric セクション（時間/セット/総重量）のフォントが小さすぎる | 中 | 視認性 |
| FAB の位置が BottomNav と近すぎて誤タップの恐れ | 中 | 操作性 |
| HistoryCard に「再利用する」「同じメニューで始める」ボタンがない | 高 | コア機能欠如 |

**佐藤（UA）の要望:**
> Strong みたいに履歴からワンタップでそのワークアウトを複製して始められないと、毎回同じメニューを組み直す地獄になる。「前回のプッシュデーと同じメニューで」が最も頻繁なユースケースなのに。

### 2-3. ワークアウト編集 (`/workouts/edit`)

| 問題 | 深刻度 | カテゴリ |
|------|--------|----------|
| 「ブロック」の概念説明がゼロ。何のためにブロック分けするのか不明 | 高 | UX/オンボーディング |
| タイトルの Pencil アイコンが小さすぎてタップターゲットとして認知しづらい | 中 | 操作性 |
| 「メモを追加…」が何もない状態のプレースホルダーだが、実際にメモ入力UIがない | 中 | 誤解を招くUI |
| 保存ボタンが種目追加後にのみ表示される。空ワークアウトを下書き保存できない | 中 | 柔軟性 |
| ブロック削除、種目削除、種目の並べ替えがすべて不可能 | 高 | 機能欠如 |
| ブロック名が「ブロック 1」「ブロック 2」固定で変更不可 | 中 | 柔軟性 |
| 保存後の遷移先が `/workouts` だが、保存成功のフィードバックが Toast のみ | 低 | UX |
| 同じ種目を同じブロックに2回追加できてしまう | 低 | バリデーション |

**中村（TA）の提案:**
> ブロックは上級者向けの概念。初心者にはシンプルに「種目リスト」でいい。ブロックはオプション機能としてデフォルトOFFにするか、「スーパーセットを組む」のような具体的な用途で導入すべき。

| 4タブ（ホーム/撮影/ワークアウト/ボディ）で「動画」が漏れている。ダッシュボードからしかアクセスできない | 高 | 情報アーキテクチャ |

## 第3部: クロスカット分析（横断的問題）

### 3-1. データ永続化の完全欠如

**全員一致で最大の問題として指摘。**

> **高橋（UC）**: すべての保存処理が `console.log` で終わっている。ワークアウト記録もボディログも動画メタデータも、ブラウザを閉じたら全消失。これは「アプリ」ではなく「インタラクティブモックアップ」。ローカルストレージ、IndexedDB、あるいは Supabase のようなBaaSでもいいから、最低限の永続化が必要。

影響範囲:
- `workouts/edit/page.tsx` — handleSave → `console.log`
- `body/BodyLogForm.tsx` — handleSave → `console.log`
- `capture/meta/page.tsx` — handleSave → `console.log`
- `videos/[id]/page.tsx` — handleMemoSave → `console.log`
- `settings/page.tsx` — handleSave → `console.log`

### 3-2. 導線設計の断絶

```
ユーザーの主要フロー（想定）:
1. ジムに到着 → アプリ起動 → 今日のメニュー確認 → 記録開始 → セット完了ごとに記録 → 完了
2. フォーム確認 → 撮影 → スロー再生 → 補助線で確認 → メモ記録

現状のフロー（実際）:
1. ダッシュボード(何もできない) → ワークアウトタブ → FAB → ゼロから作成 → ... → 保存(消える)
2. 撮影タブ → 録画 → meta入力 → 保存(消える) → 動画一覧(ダッシュボードから)
```

**山田（PM）:**
> フロー1とフロー2が完全に分離しているのが問題。「ワークアウト中に気になったフォームを撮影して、そのセットに紐付ける」というシームレスな体験がない。撮影とワークアウトが別タブ・別フローなのは、ユーザーにコンテキストスイッチを強いている。

### 3-3. デザイントークンの不徹底

| 項目 | 場所A | 場所B | 不整合 |
|------|-------|-------|--------|
| Stepper デザイン | `workout/Stepper.tsx`（`bg-chip` 丸角） | `capture/meta MetaStepper`（`border` + 角型） | 同じ機能で全く異なる見た目 |
| ボタン高さ | PrimaryRecordButton: `min-h-[40px]` | 設定保存: `min-h-[44px]` | 統一されていない |
| カード角丸 | HistoryCard: `rounded-2xl` | 設定カード: `rounded-[18px]` | 微妙に異なる |
| 種目カテゴリチップ | MovementListView: `rounded-full` + `bg-inverse`（選択時） | HistoryCard: `rounded-full` + `bg-chip` | 選択状態の表現が不統一 |
| ヘッダー構造 | Dashboard: `<p>FormCheck</p><h1>今日もいこう</h1>` | Workouts: `<p>Workouts</p><h1>ワークアウト履歴</h1>` | パターンは同じだが英語/日本語混在の必然性が不明 |

### 3-4. アクセシビリティ

| 問題 | 深刻度 |
|------|--------|
| コントラスト比: `text-muted (#9a9a9a)` on `bg-surface (#f3f3f3)` = 2.6:1。WCAG AA 基準 4.5:1 を大幅に下回る | 高 |
| コントラスト比: `text-secondary (#7a7a7a)` on `bg-white` = 4.5:1。ギリギリ AA だが AAA は不合格 | 中 |
| Stepper の `aria-label` が「減らす」「増やす」のみで、何の値を操作しているか不明 | 中 |
| BottomSheet に `role="dialog"` と `aria-modal` がない | 中 |
| 動画プレイヤーのキーボード操作が一切考慮されていない | 中 |
| フォーカストラップがモーダル/シートで未実装 | 中 |

### 3-5. パフォーマンス懸念

| 項目 | 問題 |
|------|------|
| BodyDetail の SVG チャート | データポイントが増えると DOM ノードが爆発する。Canvas への移行が必要 |
| ワークアウト編集の State Machine | 全ビュー（editor/list/detail）が同一コンポーネント内で管理。遷移時に不要な再レンダーが発生 |
| 動画ページの DrawingCanvas | 全 shapes を state で管理。描画が増えると再レンダーコストが線形増加 |
| `MOCK_WORKOUT_HISTORY` のフィルタ | 毎レンダーで全件走査。データ量が増えるとリスト表示が重くなる |

## 第4部: パネルディスカッション

### 議題1: 「このアプリの核は何か」

**山田（PM）**: このアプリは「FormCheck」と名乗っている。フォームチェックが核なのか、ワークアウト記録が核なのか、どちらかに絞るべき。両方中途半端になっている。

**中村（TA）**: フォームチェック（動画撮影＋スロー再生＋補助線）は差別化になりうる。ワークアウト記録は Strong、JEFIT、GymBook と正面衝突。勝てない。

**田中（D）**: 同意。フォームチェックを軸にするなら、ワークアウト記録はサブ機能としてミニマルにとどめ、撮影→解析→フィードバックのループを磨くべき。

**木村（TB）**: フォームチェックに振るなら、AI姿勢推定（MediaPipe Pose）くらいないと差別化にならない。補助線だけでは弱い。

**結論**: **フォームチェック特化 + ミニマル記録** が現実的なポジショニング。ワークアウト記録で Strong と戦うのは得策でない。

### 議題2: 「初心者と上級者、どちらを狙うか」

**鈴木（UB）**: 今の設計は初心者に厳しい。ブロック概念も種目名もマニア向け。

**伊藤（TC）**: 初心者に寄せるなら、おすすめメニュー提案、種目の動画デモ、ステップバイステップのガイドが最低限必要。

**佐藤（UA）**: 上級者に寄せるなら、テンプレート、スーパーセット、1RM追跡、プログレッシブオーバーロードの自動提案が必要。

**山田（PM）**: TAM（Total Addressable Market）から考えると、初心者〜中級者（筋トレ歴3年以下）がボリュームゾーン。ただしフォームチェックのニーズは「自分のフォームが正しいか不安な層」なので、初心者〜中級者に刺さる。

**結論**: **筋トレ初心者〜中級者（自分のフォームに不安がある層）** がメインターゲット。

### 議題3: 「BottomNav の構成は適切か」

**田中（D）**: 現在の4タブ「ホーム/撮影/ワークアウト/ボディ」に違和感がある。「撮影」がメイン導線にあるのは、フォームチェック特化なら正しいが、同時にボディログがメインタブにある必要があるか疑問。

**山田（PM）**: 提案として:

```
案A（フォームチェック軸）:
ホーム | 撮影 | ワークアウト | ライブラリ | 設定

案B（記録アプリ軸）:
ホーム | ワークアウト | 撮影 | ボディ | 設定

案C（現状維持改善）:
ホーム | ワークアウト | [+](撮影/記録) | 動画 | プロフィール
```

**高橋（UC）**: 案Cの中央FABパターン（Instagram/TikTok式）が一番しっくりくる。「記録する」というアクションを中央に置いて、左右にコンテンツ参照系を配置する。

**結論**: 案Cが有力。ただし実装前にユーザーテスト（カードソーティング）で検証すべき。

### 議題4: 「このままローンチ可能か」

全員の評価:

| 参加者 | 判定 | 理由 |
|--------|------|------|
| 田中（D） | **不可** | デザインシステムが未確立。トークン不整合が多すぎてスケールしない |
| 佐藤（UA） | **不可** | データが消える。使う意味がない |
| 鈴木（UB） | **不可** | 初心者として何をすればいいかわからない。離脱する |
| 高橋（UC） | **不可** | sessionStorage依存、console.log保存。技術的にプロダクションレベルでない |
| 山田（PM） | **不可** | MVPの定義を満たしていない。記録→保存→振り返りのループが成立しない |
| 中村（TA） | **不可** | テンプレートなしでクライアントに渡せない |
| 木村（TB） | **不可** | データの粒度と分析機能が不足。使い続けるモチベーションが生まれない |
| 伊藤（TC） | **不可** | 初心者向けコンテンツ（画像、動画、ガイド）が皆無 |

**全会一致: ローンチ不可。**

## 第6部: 総括

### 評価されたポイント（ポジティブ）

1. **ビジュアルの統一感**: モノクロ基調 + Inter/Noto Sans JP の組み合わせは洗練されている
2. **フォームチェックのコンセプト**: 補助線オーバーレイ、スロー再生、描画ツールは筋トレアプリとしてユニーク
3. **日本語ローカライゼーション**: 種目の説明文が自然な日本語で、カタカナだけでなく意味が伝わる
4. **コンポーネント分割**: Toggle, Stepper, Toast, PrimaryRecordButton など再利用可能なパーツが設計されている
5. **アニメーション**: slide-in/slide-back/fade-in による画面遷移が滑らかで、ネイティブアプリライク

### 致命的な問題（ネガティブ）

1. **データが一切保存されない** — 使用する意味がない
2. **テンプレート/ルーティン機能がない** — 毎回ゼロから作成は非現実的
3. **ワークアウト実行モード（ライブトラッキング）がない** — 記録アプリの根幹が欠落
4. **初心者向けコンテンツ（イラスト、ガイド、オンボーディング）がゼロ**
5. **フォームチェックとワークアウト記録のフローが分断** — コアバリューが活きない
6. **種目数が18種のみ** — アプリとして最低100種は必要
7. **CRUD の D（削除）と U（更新）が全画面で未実装**
8. **アクセシビリティがWCAG AA基準を満たしていない**

### 最終結論

> **現状はインタラクティブプロトタイプとして高品質だが、プロダクトとしてローンチ可能な状態ではない。**
>
> Phase 0（基盤整備）と Phase 1（コアループ完成）を最低限クリアした上で、限定的なクローズドベータとしてリリースし、実ユーザーのフィードバックを収集することを推奨する。
>
> フルローンチ（App Store / Google Play 公開レベル）には Phase 0〜3 の完了が必要であり、現実的な見積もりで **約2〜3ヶ月の開発期間** が必要。
