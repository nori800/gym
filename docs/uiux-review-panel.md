## 第0部: 直近アップデートの要約（2026-04-22 時点）

以下は**本レビュー掲載時点のコードベース**に反映済みの改善である。

| 領域 | 内容 |
|------|------|
| **種目詳細** | ウェイトモード／アシスタンスを**デフォルト折りたたみ**のカード化。初回表示でセット＆レップへ素早く到達し、上級オプションは展開時のみ表示。 |
| **BottomNav** | **5タブ構成**（ホーム・ワークアウト・**中央撮影**・動画・ボディ）。中央は円形FAB相当で視覚的に強調し、フォームチェックをコア導線として明示。 |
| **動線** | ワークアウト編集から撮影へのCTA、ダッシュボードの空状態ガイド、カスタム日付ピッカー等は先行イテレーションで整備済み。 |
| **誠実な設定** | 効かないテーマ切替を削除し、「通知＝準備中」のみ明示。 |
| **用語** | ウェイトモード／アシスタンスをジム一般的な説明（マシン専用語の排除）に寄せた。 |

**未解決（CRUD・バックエンド前提の課題は別枠）**: データ永続化、履歴詳細、テンプレート、撮影とセットの自動紐付け、種目サムネ実装など。

## 第1部: ファーストインプレッション（改善後の所感）

### 田中（D）— デザイナー

> デザイントークンとカード階層はかなり揃った。特に種目詳細で「基本情報 → ウォームアップ → **折りたたみの詳細オプション** → セット」という視線の流れは、Tonal由来の密度を保ちつつ初見の負荷を下げている。BottomNav の中央撮影は「このアプリの主役は何か」を一目で言語化できていて良い。

**残課題:** 種目ヒーローがプレースホルダのままなので、第一印象の「展示品感」はまだ抜けきらない。

### 佐藤（UA）

> 種目追加の画面、最初は長かったけど、いまはデフォルトでオプションが畳まれてるから「まず回数と重量だけ決めて追加」がやりやすい。下のナビでカメラが真ん中にデカくあるのは分かりやすい。動画タブも横にあって迷子になりにくい。

### 鈴木（UB）

> 「ウェイトモード」って言葉は最初ビビるけど、閉じたままだと中身が見えなくて逆に安心。タップしたら説明が出るから、自分のペースで開けるのはいい。撮影が真ん中にあると「フォーム見るアプリなんだ」って伝わる。

### 高橋（UC）

> ナビ5分割＋中央FABはよくあるパターンで実装も素直。`pathname.startsWith` 系のマッチで子ルートも拾えているなら運用しやすい。永続化とモックの壁は相変わらず。

### 山田（PM）

> IA 上「撮影＝コア」が BottomNav で一発で伝わるようになったのはプロダクトメッセージとして大きい。それでも「ワークアウト記録」と「撮影」のデータ結合は未実装なので、体験の物語はまだ途切れる。

### 中村（TA）

> 折りたたみはクライアント向けにも好印象。多くの人はノーマル＋自力で十分。ドロップセットや補助ありを選ぶ層だけ開けばいい。

### 伊藤（TC）

> 初心者は「開かなくていい項目」が減ると迷いが減る。中央の撮影はジムでスマホ見せられても「フォーム撮るやつです」と説明しやすい。

## 第3部: クロスカット（現状の位置づけ）

### 3-1. データ永続化

**変更なし（最大リスク）:** コア保存は多くがモック／`console.log` 前提。ローンチ判断では引き続き **Phase 0 必須**。

### 3-2. 導線の物語

ワークアウト編集から撮影へのCTAは追加済みだが、**セットIDとの自動紐付け**は未実装。フロー統合の完成は CRUD ＋状態設計が必要。

### 3-3. アクセシビリティ

コントラスト・フォーカストラップ・`aria-*` の強化は先行イテレーションで進展。`muted` トークンはキャプション用途のまま監査では注意点になりうる。

## 第4部: 定量めいた総合スコア（プロトタイプ前提）

CRUD・本番永続化を**意図的に除外**した「UI/UX プロトタイプ」としての評価。

| カテゴリ | スコア | コメント |
|----------|--------|----------|
| デザインシステム統一 | A | トークン・角丸階層・カード行の一貫性が高い。 |
| 動線・IA | A− | 中央撮影＋5タブでコアが明確。データ結合は次フェーズ。 |
| 初心者負荷 | A− | 折りたたみで種目詳細の初回負荷が下がった。種目画像は未着手。 |
| Tonal 準拠（参照UI） | A | ブロック／ステッパー等の構造模倣は継続して高水準。 |
| アクセシビリティ | A− | 改善継続中。カスタム日付等のキーボード操作は更に詰められる。 |
| **プロトタイプ総合** | **A** | デモ・社内合意・デザインQAには十分耐える。 |

**プロダクトとしてのローンチ可否:** データ永続化とコアループ未完成のため **「ストア公開」は不可** の判断は変わらず。クローズドβは Phase 0 完了後。

## 第6部: 総括

### ポジティブ（今回の文書更新で強調）

1. **種目詳細の情報設計**: 詳細オプションのプログレッシブディスクロージャー化により、初級者の「長い画面の圧」が緩和された。  
2. **BottomNav の意味づけ**: 5タブ＋中央撮影FABにより、「FormCheck＝フォームを撮って改善する」というプロダクト仮説がナビゲーション単体で説明できる。  
3. **先行イテレーションの積み上げ**: デザイントークン、オンボーディング、ブロック説明、日付UI、設定の誠実さなど、プロトタイプ品質は継続的に向上している。

### ネガティブ（根強い課題）

1. **永続化と CRUD** — ユーザー価値の根幹。  
2. **種目ビジュアル不在** — サムネ／イラストがない限り、初心者体験は頭打ち。  
3. **ワークアウト実行中のライブ体験**（レスト、セット完了フロー）— 未実装。  
4. **撮影と記録のデータ統合** — 導線は増えたが、まだ「別アプリ二つ」に近い。

### 最終結論

> **UI/UX プロトタイプとしては A 帯に到達。** 特に種目詳細の折りたたみと BottomNav の中央撮影は、ターゲット（フォームに不安のある初〜中級者）へのメッセージと操作負荷の両面で効いている。  
>  
> 一方で **プロダクトとしてのローンチ** は、引き続き Phase 0（永続化・CRUD）なしには判断できない。本ドキュメントは「現時点の画面仕様」に合わせて更新され、古い指摘（4タブのみ・種目詳細の常時全展開など）は**解消済みまたは緩和済み**として差し替えた。
