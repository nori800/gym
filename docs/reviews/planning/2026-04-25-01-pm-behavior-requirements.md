## 1. 目的

「会員用アプリにトレーナー権限が足された」状態から脱し、**行動（ジョブ）単位**で会員とトレーナーに必要な能力を定義し、**現状で足りないもの**を洗い出す。

---

## 2. 大前提: マンツーマン型トレーニングジム

本プロダクトの想定する現場は **マンツーマンのパーソナルトレーニングジム**（担当トレーナーが常に主役の関係）であり、**一般ジムの会員が自走するソロ記録アプリ**ではない。

**オペレーション分担（確定）**: **トレーナー口座の開設・`role=trainer` 付与は店舗オーナー（私）が実施**する。**会員の担当紐付け（招待）は担当トレーナーが `/trainer` から実施**する。詳細は [2026-04-25-flow-trainer-to-member-onboarding.md §0](./2026-04-25-flow-trainer-to-member-onboarding.md) を参照。

### 2.1 現行案・画面が陥りやすい誤解

- UI は **会員がホームから撮影・ワークアウト・体組成を自分で回す**導線が中心になっている。
- そのまま読むと「**会員が全部自分でやる**」前提に見える。マンツーでは実際には次のような割合が普通である:
  - **セッション中**: トレーナー主導でメニュー・セット管理・フォーム確認が進む（端末はトレーナー手元、または二人で見る）。
  - **セッション間**: 宿題（自重・有酸素・フォーム動画の提出など）を **トレーナーが指示 → 会員が実行・記録** する。
- つまり **会員の「自分でやる」は主目的ではなく、指示された範囲での実行・証跡の提出**に近い。現アプリはこの **「誰が主導したか」** が画面に出てこない。

### 2.2 プロダクト上の修正方針（要件レベル）

| 観点 | あるべき姿（マンツー） | 現状とのギャップ |
|------|------------------------|------------------|
| 主導 | トレーナーが **今日の宿題・次回までの課題** を見える化し、会員はそれに沿って撮影・記録 | 「宿題」「依頼」概念がなく、会員が任意に撮るだけ |
| ホーム（会員） | **担当からの未返信・次にやること** が上に来る | 自分の週次サマリーだけ（ソロ向け） |
| ホーム／ハブ（トレーナー） | **今日の担当セッション・未レビュー動画** が最優先 | 自分のダッシュと同型 |
| 記録 | セッション中は **トレーナー入力 or 共同確認** も許容する設計を検討 | 会員 CRUD のみ |
| 撮影 | 「**トレーナーから撮って提出依頼**」があれば会員の自己主導と矛盾しない | 依頼なしの自由撮影のみ |

上記は **未実装の方向性**であり、次スプリントで全部やる必要はない。ただし **バックログの優先度付け**では「ソロ会員アプリ」ではなく **マンツー前提**で欠損を読むこと。

## 3. ペルソナと主ジョブ

### 3.1 会員（メンバー）

| ジョブ（行動） | 成功状態 |
|----------------|----------|
| J-M1 今日やることを把握する | 直近のセッション・体重・動画の有無が一目で分かる |
| J-M2 トレを記録する | テンプレから素早く入力し、履歴に残る |
| J-M3 フォームを撮る | 撮影→メタ入力→自分のライブラリに載る |
| J-M4 過去の自分と比較する | 同一種目の動画比較や履歴参照ができる |
| J-M5 体組成を追う | 体重等を時系列で見・記録できる |
| J-M6 担当トレーナーに状況を渡す | 招待・紐付けのうえ、**指示された範囲**（宿題の動画・記録）がトレーナーに届く（会員は「全部自分で設計」しない） |

### 3.2 トレーナー

| ジョブ（行動） | 成功状態 |
|----------------|----------|
| J-T1 メンバーを増やす・外す | 招待・紐付け解除が迷わずできる |
| J-T2 「今日誰を見るか」決める | 新着・未レビュー等で優先順位が付く（または一覧で即断できる） |
| J-T3 メンバーの動画でフォーム判断する | ストリーミングまたは共有URLで確実に再生できる |
| J-T4 メンバーの記録を俯瞰する | ワークアウト・体重推移を横断で把握できる |
| J-T5 介入する | 動画やセッションに対しコメント・指示が残せ、メンバーが気づける |
| J-T6 自分もトレする | 会員と同様に自分の記録・撮影ができる（**副次**。店舗スタッフ兼用時のみ主役になりうる） |

## 4. ジョブ × 現状 × 不足（要件マトリクス）

凡例: ✅ おおむね満たす / ⚠️ 部分的 / ❌ 未充足

### 会員

| ジョブ | 現状（v0.5.0） | 不足・要件 |
|--------|----------------|------------|
| J-M1 | ✅ `/dashboard` で本人サマリー | トレーナーからの「未読フィードバック」等はなし（通知・コメント基盤なし） |
| J-M2 | ✅ ワークアウト CRUD | エラー一部がユーザーに見えない（親レポート UX-N4）。**トレーナー指示と紐づく記録**の概念なし（マンツーではギャップ） |
| J-M3 | ✅ 撮影〜保存 | Safari/WebM 等の別要件。**「依頼された撮影」**と自由撮影の区別なし → 会員がソロで回している印象に繋がる |
| J-M4 | ✅ 比較・詳細アノテ | 描画／ガイドの役割が分かりにくい（情報設計） |
| J-M5 | ✅ `/body` | 特になし（会員経路） |
| J-M6 | ⚠️ RLS＋`trainer_id` で閲覧可能だが、**共有URLでは動画不可** | **共有先でも動画再生（P0）**。「担当が見る」文脈の UI（未読フィードバック等）が弱く、**依頼→提出→レビュー**の一本線が見えない |

### トレーナー

| ジョブ | 現状（v0.5.0） | 不足・要件 |
|--------|----------------|------------|
| J-T1 | ⚠️ `/trainer` で招待。検索が display_name 中心 | **メールベース招待**（親 UX-N3）、失敗時のユーザー向けフィードバック強化 |
| J-T2 | ❌ ホームは本人サマリーのみ。`/trainer` は一覧＋件数 | **トレーナー向け「今日のレビュー」／新着キュー**、またはデフォルト着地の再設計 |
| J-T3 | ⚠️ アプリ内 `?member=` で再生可の想定だが、**共有ページは動画なし** | **signed URL ＋ 共有UI（P0）** |
| J-T4 | ⚠️ メンバーごとにページ遷移は可能だが横断サマリーなし | **トレーナー進捗ダッシュボード（P1）**（週次・体重・ボリューム等） |
| J-T5 | ❌ 動画へのコメント・タイムスタンプ指示なし | **フィードバックデータモデル＋UI（P1）**、通知（P2）とセットで価値が立つ |
| J-T6 | ✅ 会員と同じ画面で記録可能 | マンツー主戦場では副次。**モードの見せ分け**で「店舗のお客」と「スタッフ私用」を混ぜない |

## 5. 不足分の優先パッケージ（プロダクト判断）

| パッケージ | 含む要件 | 理由 |
|------------|----------|------|
| **P0 共有の完成** | 共有ページ動画再生、API の堅牢化（admin client） | コア価値「フォームを見る」が共有経路で死んでいる |
| **P1 トレーナー価値の核** | フィードバック、トレーナー進捗ビュー、ホーム／ナビのロール分離、**（中長期）依頼・宿題・セッション文脈** | 課金理由の主戦場。マンツーでは「会員の自力運用」だけでは価値が説明しづらい |
| **P1 信頼・運用** | エラー表示の統一、FocusTrap、残存 try/catch | 離脱・クレーム抑制 |
| **P2 拡張** | PDF 配線、通知、PWA、招待UX詳細、テスト拡大 | 差別化・継続率 |

## 7. PMメモ（意思決定の保留）

- **トレーナーのデフォルト着地**を `/dashboard` のままにするか `/trainer` にするかは、デザイン・実装とセットで A/B しやすいようフラグ化を検討。
- **メンバーがトレーナーのフィードバックをどこで読むか**（動画詳細／通知／インボックス）は P1 設計時に一度決める必要あり。
