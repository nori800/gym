## 2. 全体総評（結論先出し）

### スコア: 5.2 / 10（前回 2.9 → +2.3）

**「骨格は組み上がったが、壁も屋根もスカスカ。」**

前回の「ガワだけ」状態からは確実に前進した。データが消えない、エラーが表示される、CRUDができる——これは大きな進歩。だが、**今回の修正で新たに生まれたセキュリティホールと設計矛盾が深刻**で、「有料βテスト可能」(7.5) という前回の修正報告の自己評価は**過大**。

致命的な問題:
1. **RLS（Row Level Security）がトレーナー機能と完全に矛盾** — トレーナーが会員データを見れない
2. **共有リンクのRLSが全データ漏洩リスク** — `SELECT`が条件なしで全行読み取り可能
3. **PWAアイコンが存在しない** — manifest参照切れ
4. **APIルートにスキーマ不整合** — `workout_logs` テーブルがDBに存在しない
5. **ミドルウェアが `/trainer` を保護していない**

「機能を追加した」だけで「正しく動く」保証がないコードが大量にある。

## 3. ペルソナ別評価

### 3.1 トレーナー視点（4名）

#### トレーナーA（パーソナルジム経営・10年目）

> **「会員管理画面が真っ白になった。使い物にならない。」**

`/trainer` ページは存在するが、**RLSポリシーが `auth.uid() = user_id` のみ**のため、トレーナーが会員の `profiles` / `workouts` / `videos` を取得できない。UI上は永遠にスピナーが回るか、空のリストが表示される。

招待機能も `profiles` の `ilike` クエリが RLS で弾かれるため、**会員を見つけることすらできない**。

**評価**: 1/10 — 機能として成立していない

#### トレーナーB（フリーランストレーナー・5年目）

> **「テンプレートは嬉しいが、会員に共有できないなら半分の価値。」**

テンプレート保存/読み込みは自分用には機能する。だがトレーナーが作成したテンプレートを会員に配布する導線がない。`workout_templates` のRLSも `auth.uid() = user_id` で、**他者への公開手段がない**。

**評価**: 4/10 — 個人用としては使えるが、トレーナーの本質的ニーズ未達

#### トレーナーC（スポーツジムチーフ・7年目）

> **「共有リンクからの動画閲覧、結局できなかった。」**

`/api/share` の GET は `createServerSupabaseClient()` を使ってクエリするが、共有リンクを開くのは**未認証のユーザー**。`videos` テーブルのRLSは `auth.uid() = user_id` なので、**動画のメタデータすら返らない**。StorageのRLSも同様。共有リンク機能は**設計段階で破綻**している。

**評価**: 2/10

#### トレーナーD（ボディメイクコーチ・3年目）

> **「角度計測、面白い。でもタップ精度が悪すぎて使えない。」**

`AngleTool` は3点タップで角度を計測するが、動画再生中のオーバーレイ上でタップすると**動画のシーク操作と競合**する。`active` 時のみ反応するとはいえ、ツール切り替えの導線が直感的でない。モバイルの細い指では3点を正確に置くのが困難。

**評価**: 5/10 — コンセプトは良いが実用性に課題

### 3.2 ジム初心者〜中級者視点（4名）

#### ユーザーE（ジム歴3ヶ月・初心者）

> **「前回の記録が見れるのは最高。でも推移グラフがバグってる。」**

`LastSessionBadge` は種目選択時に前回の重量・回数を表示し、**これは非常に嬉しい改善**。ただし `ProgressGraph` のボリュームモード (`volume`) で単位が `kg` と表示される——ボリュームは `kg × 回 × セット` なので単位としておかしい。

また `useLastSession` は一度キャッシュすると**新しいワークアウトを保存しても値が更新されない**（モジュールレベルの `Map` キャッシュが陳腐化）。

**評価**: 6/10

#### ユーザーF（ジム歴1年・中級者）

> **「動画比較、期待してたけど同期再生のタイミングがズレる。」**

`/videos/compare` の同期再生は `v.play()` を `Promise.all` で呼んでいるが、`play()` は**動画バッファリング完了まで待たない**。ネットワーク状況で片方だけ先に再生開始し、同期が崩れる。`v.play().catch(() => {})` で再生失敗を無視しているのも問題。

**評価**: 5/10

#### ユーザーG（ジム歴6ヶ月・初心者）

> **「ゲストで撮影→保存したら『保存しました』って出たけど、次回開いたら消えてた。」**

`capture/meta/page.tsx` のゲスト保存は Supabase アップロードをスキップしつつ `setPhase("saved")` まで到達する。成功チェックマーク付きの「保存しました」画面を見せておきながら、**実際にはどこにも保存されていない**。これは前回指摘の `console.log("save (guest)")` と本質的に同じ問題の**見た目を変えただけ**。

**評価**: 3/10 — 嘘の成功体験を提供している

#### ユーザーH（ジム歴2年・中級者）

> **「休憩タイマー、BottomNavと被って使えない。」**

`RestTimer` のコンパクトモードは `fixed bottom-24` に配置されるが、`BottomNav` と**重なるスクリーンが存在する**。BottomNavの高さ計算と `safe-area-inset-bottom` の関係で、端末によっては完全に隠れる。

**評価**: 5/10

### 3.3 PM視点（1名）

#### PM（プロダクトマネージャー）

> **「セキュリティレビューを通さずにトレーナー機能と共有リンクを出荷したら訴訟モノだ。」**

**最大の懸念はセキュリティ**:

3. `/api/share` GET が参照する `workout_logs` テーブルが**マイグレーションに存在しない**。サーバーエラーが出る。

4. `export.ts` の `generatePDFHTML` がユーザー入力を**エスケープなしでHTMLに埋め込む** — XSSリスク。

> **リリース判断: NG。セキュリティ修正なしではβすら不可。**

**評価**: 4/10

## 5. セキュリティ・RLS の致命的問題

### 5.1 共有リンクのトークン漏洩（Critical）

```sql
-- 003_extended_tables.sql
CREATE POLICY "shared_links_public_read" ON public.shared_links
  FOR SELECT USING (expires_at IS NULL OR expires_at > now());
```

この RLS ポリシーは **認証状態を問わず、有効期限内の全行を SELECT できる**。つまり任意のクライアントが `supabase.from("shared_links").select("*")` を叩くと、**全ユーザーの共有トークン・紐付き動画ID・ワークアウトIDが取得できる**。

正しくは `USING (token = current_setting('request.header.x-share-token', true))` のような**トークンベースの行制限**か、API経由のみのアクセスに制限すべき。

### 5.2 トレーナー機能の RLS 矛盾（Critical）

現行:
- `profiles`: `auth.uid() = user_id`
- `workouts`: `auth.uid() = user_id`
- `videos`: `auth.uid() = user_id`

必要:
- トレーナーが `trainer_id = auth.uid()` のプロフィールとその所有データを読める追加ポリシー

```sql
-- 必要な追加ポリシー（例）
CREATE POLICY "trainer_read_members" ON public.profiles
  FOR SELECT USING (auth.uid() = user_id OR trainer_id = auth.uid());

CREATE POLICY "trainer_read_member_workouts" ON public.workouts
  FOR SELECT USING (
    auth.uid() = user_id
    OR user_id IN (SELECT p.user_id FROM public.profiles p WHERE p.trainer_id = auth.uid())
  );
```

### 5.3 video_annotations の他人汚染（High）

`video_annotations` のRLSは `auth.uid() = user_id` だが、**`video_id` の所有者チェックがない**。他ユーザーの動画IDが分かれば（共有リンク等で推測可能）、その動画にアノテーション行を INSERT できる。

### 5.4 API ルートの `workout_logs` 不整合（High）

`/api/share` GET が `.from("workout_logs")` を参照するが、DBマイグレーションに `workout_logs` テーブルは**存在しない**。正しくは `workouts` テーブル。このAPIは常に500エラーを返す。

### 5.5 PDF エクスポートの XSS（Medium）

`generatePDFHTML(title, content)` がHTMLテンプレートに**エスケープなしで文字列を埋め込む**。ユーザー入力（ワークアウトタイトル等）がそのまま `<h1>${title}</h1>` に入るため、`<script>` を含むタイトルでXSSが成立しうる。

### 5.6 ミドルウェアのフェイルオープン（Medium）

環境変数未設定時・`getUser` 失敗時に `NextResponse.next()` で通過させるため、**一時障害時に認証ガードが完全に無効化**される。セキュアデフォルトの原則に反する。

### 6.2 エラーハンドリングの抜け

| ファイル | 問題 |
|---------|------|
| `videos/page.tsx` | 動画一覧取得の `.then()` で `error` を見ていない。失敗時ローディングが永続化 |
| `body/page.tsx` | `fetchLogs` で `error` を見ていない。失敗時に誤った空表示 |
| `settings/page.tsx` | `ProfileForm` の初回取得で `error` 未処理 |
| `trainer/page.tsx` | `fetchRole`/`fetchMembers` の Supabase エラー未チェック。`role === null` 永続スピナー |
| `useAuth.ts` | `getUser().then()` で `error` 分岐なし。ネットワークエラーで `loading` が `true` のまま固まる |
| `compare/page.tsx` | `v.play().catch(() => {})` で再生失敗を無視 |

### 6.3 CustomGuides のドラッグ不能バグ

`CustomGuides.tsx` で `draggingRef.current` をチェックして `pointerEvents` を切り替えるが、**ref の更新は再レンダーをトリガーしない**。ドラッグ開始直後に `pointer-events: none` のままになり、**実質ドラッグ操作ができない**可能性が高い。

### 6.4 useLastSession のキャッシュ陳腐化

モジュールレベルの `Map` キャッシュは `fetchedRef` で重複フェッチを防止するが、**新しいワークアウトを保存した後もキャッシュが更新されない**。ユーザーが種目を追加するたびに古い「前回の記録」が表示される。

### 6.5 ゲスト保存の欺瞞

`capture/meta/page.tsx` のゲスト時:
1. チェックマーク付き「保存しました」画面を表示
2. 実際にはどこにも保存されていない
3. 「もう1セット撮影」ボタンまで表示

前回の `console.log("save (guest)")` を**UIで隠しただけ**。ユーザーに誤った成功体験を与える設計は信頼を破壊する。

### 6.6 Tailwindアニメーションの二重定義

`globals.css` と `tailwind.config.ts` の両方に `slide-in` / `slide-back` / `fade-in` / `sheet-up` のキーフレームが定義されている。しかも**中身が異なる**（config: `translateX(100%)` vs globals: `translateX(40%)` 等）。CSS読み込み順で結果が変わり、デバッグが困難。

### 6.7 PWAアイコンが存在しない

`manifest.json` と `layout.tsx` の `metadata.icons` が `/images/icon-192.png` / `/images/icon-512.png` を参照するが、`public/images/` に**PNGファイルが1つも存在しない**。PWA インストール時にアイコンが表示されない。

### 6.8 mocks ディレクトリ名

`lib/mocks/movements.ts` は本番でも使う静的マスタデータだが、パスが `mocks` のまま。「モック = ダミーデータ」の意味合いで誤解を招く。`lib/data/movements.ts` や `lib/master/movements.ts` が適切。

### 6.9 Error Boundary の不在

`app/layout.tsx` に React Error Boundary がない。子コンポーネントの未捕捉例外がアプリ全体をクラッシュさせる。Next.js の `error.tsx` ファイルも `app/` 直下に存在しない。

### 9.3 一言まとめ

**「骨は組み上がったが、鍵をかけ忘れたドアだらけ。」**

v0.1.0 の「ガワだけ」からは明確に前進した。データが保存され、エラーが表示され、CRUDが動く。種目マスタの充実、テンプレート、推移グラフなど「使いたい」と思わせる機能が増えた。

しかし、**機能追加のスピードにセキュリティ設計が追いついていない**。

- トレーナー機能は **RLS で完全に動作不能**
- 共有リンクは **トークン漏洩リスク**
- PWA は **アイコン不在で不完全**
- テストは **0件**
- God コンポーネントは **前回より悪化**

前回の修正報告が自己評価「7.5/10」としていたのは**過大評価**。実態は 5.2 がフェアな数字だ。

**最優先アクション**: P0 のセキュリティ6項目を即座に修正すること。RLS ポリシーの矛盾を解消し、`shared_links` のトークン漏洩を塞ぎ、ミドルウェアのフェイルオープンを閉じる。これなしでは内部テストすら安全に実施できない。

その上で P1 のエラーハンドリング抜け・バグ修正を完了させれば、「無料β可能」(6.0) に到達する。
