## 3. v3→v4 の変更サマリ

### 良い変更

1. **セキュリティ強化**: `/api/share` POST に `video_id`/`workout_id` 所有者チェック追加。GET に service role 使用で匿名閲覧可能に
2. **テスト基盤**: Playwright 導入、5テストファイル計18ケース（10パス、8スキップ）
3. **God component 分割**: `workouts/edit/page.tsx` 727→461行。`useWorkoutPersistence` フック抽出
4. **アクセシビリティ**: CustomGuides に `role="slider"` + キーボード操作。AngleTool に Escape/Ctrl+Z。タブにロービング tabIndex
5. **DB hardening**: GRANT EXECUTE 明示、search_path 修正

### 懸念のある変更

1. **`sanitizeHTMLContent` は不完全**: `<script>`, `on*`, `javascript:` のみ除去。`<iframe>`, `<object>`, `<meta http-equiv="refresh">`, CSS `expression()`, SVG payload 等は残る
2. **`workouts/edit` 分割は中途半端**: 461行はまだ大きい。`handleAddAndSave` が `persistence.handleSave()` を呼ぶが、draft 更新が非同期 setState のため**タイミングバグのリスク**あり
3. **E2E テストの認証テストは環境変数依存でスキップ**: 実質テストされているのは未認証フローのみ

### 4.1 修正済み項目

| shared_links 所有者チェック | ✅ POST で video/workout の user_id 一致確認 |

| GRANT EXECUTE | ✅ anon, authenticated に明示付与 |

### 4.2 新たに発見された問題

| # | 重大度 | 問題 | 箇所 |
|---|--------|------|------|
| SEC-1 | **High** | **共有リンク GET で `link.user_id` と `video.user_id` の整合性未検証**。悪意あるユーザーが自分の shared_links に他人の video_id を挿入（RLS は user_id チェックのみ）した場合、GET 側は admin クライアントで無条件取得する | `app/api/share/route.ts:127-142` |
| SEC-2 | **Medium** | **POST の `request.json()` に try/catch なし**。不正 JSON で 500 + スタック露出リスク | `app/api/share/route.ts:16` |
| SEC-3 | **Medium** | **`sanitizeHTMLContent` が不完全**。`<iframe>`, `<object>`, `<embed>`, `<svg onload>`, CSS `expression()`, `<meta http-equiv="refresh">` 等が通過する | `lib/utils/export.ts:78-84` |
| SEC-4 | **Medium** | **ミドルウェアの `/api/*` fail-open**。`updateSession` 例外時に API ルートは認証なしで通過。現状は `/api/share` のみだが拡張時に危険 | `middleware.ts:14` |
| SEC-5 | **Low** | **`.env.example` に実プロジェクトの URL/Key が記載**。公開キーだが、プロジェクト特定の足がかりになる | `.env.example:1-2` |

### 4.3 評価

v2 の致命的 P0 6件は全て解消。v3 の残存も概ね対応。しかし **SEC-1 は IDOR の亜種**で、shared_links のインサートが user_id 制約のみの場合に他人のリソースを公開できてしまう。**POST 側では防いでいるが、DB 層の CHECK 制約がない**ため、直接 Supabase クライアントから INSERT する攻撃パスが残る。
