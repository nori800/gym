## 3. セキュリティ監査

### 3.1 修正済み（良い点）

- **IDOR防止**: shared_links GET で所有者検証が入った。トークン→リンク行→video/workout取得→user_id照合の流れは設計として妥当
- **XSS防御**: sanitizeHTMLContent のブロックリスト方式は前回の正規表現3本より遥かに堅牢
- **middleware**: /api/* が fail-open から外れた。/share/ のみ例外は合理的

### 3.2 新たに発見した問題

| # | 重要度 | 内容 |
|---|--------|------|
| SEC-N1 | P1 | `createAdminClient()` が throw する場合、GET ハンドラに try/catch がない。`SUPABASE_SERVICE_ROLE_KEY` 未設定で 500 + スタック露出の可能性 |
| SEC-N2 | P2 | POST の所有権確認（40-45, 54-59行）で `.error` を見ていない。DB障害時に 403 が返り、真の原因が隠蔽される |
| SEC-N3 | P2 | `(supabase.rpc as any)` — 型安全性が破壊されており、リグレッションに弱い |
| SEC-N4 | P3 | 共有リンクにレート制限なし。トークン48文字hex で総当たりは非現実的だが、方針としてのギャップ |
| SEC-N5 | P2 | `workouts/edit` の既存読み込みに `user_id` 条件なし（RLS依存だが、エラーメッセージが汎用的で悪意あるリクエストの区別ができない） |

**結論**: 致命的な穴は塞がったが、運用堅牢性の穴が複数残る。

**スコア: 8.5/10**（v4: 7.0）

### 8.1 ¥20,000/月の判断基準（再掲）

トレーナーが月額2万円を払う条件:
1. **メンバーの動画をリモートで確認**できる
2. **フォームにフィードバック**（コメント/アノテーション）を残せる
3. **メンバーの進捗を追跡**（体重推移、ボリューム推移）できる
4. **通知**で新しい記録を即座に把握できる
5. 生徒にとっても**使いやすい**（オンボーディング、操作説明不要）

## 10. 最終結論

### v5 で良くなった点（認める）

1. **セキュリティの基盤が整った**: IDOR、XSS、middleware の3大脆弱性は実際に修正された
2. **デッドコードが減った**: AngleTool、CustomGuides、ShareLink が UI から到達可能になった
3. **エクスポートが使える**: CSV 2本は実際に動作する
4. **タッチターゲットの一括修正**: 地味だがモバイルファーストのアプリとして正しい判断

### v5 で期待に届かなかった点

1. **共有ページで動画が見られない**: アプリの名前が「FormCheck」なのに、共有先でフォームをチェックできない
2. **God component 分割が 25% で止まった**: 605行の settings/page.tsx を放置して「分割完了」と申告するのは不誠実
3. **テストが export.ts の1ファイルに集中**: 「Vitest 導入 + 18テスト」は数字として立派だが、アプリの核心（ワークアウト記録、動画管理、トレーナー連携）が全くテストされていない
4. **FocusTrap が1箇所だけ**: 「フォーカストラップ統一」と銘打ちながら10個中1個だけ対応
5. **トレーナーが金を払う理由がまだない**: フィードバック機能、通知、進捗追跡がないトレーナー向け SaaS に月2万円は払えない

### 最終スコア

| 指標 | v4 | v5 | 自己評価 | 実態 |
|------|-----|-----|---------|------|
| 総合 | 7.1 | 7.8 | 8.5+ | **7.8** |
| ¥20,000到達度 | 15% | 25% | 40%+ | **25%** |

**¥20,000には遠い。現実的には¥1,500〜2,000/月のアプリ。**

### ¥20,000に到達するために必要なこと（**AI機能はスコープ外**）

方針として **姿勢推定・自動フィードバック等のAIは採用しない**。価値は「人（トレーナー）が動画を見て判断する」ワークフローと、記録・共有・通知の完成に寄せる。

1. **動画共有の完成**（signed URL発行 + 再生UI）— +¥4,000/月（コア価値の回復）
2. **トレーナーのフィードバック機能** — コメント/タイムスタンプ付きメモ → +¥6,000/月
3. **メンバー進捗ダッシュボード**（トレーナー側の一覧・推移の明確化）— +¥4,000/月
4. **プッシュ/メール通知** — +¥3,000/月
5. **アノテーション体験の整理**（補助線・描画・ガイド・角度の情報設計とバグ修正）— +¥2,000/月
6. **PWA + オフラインキャッシュ** — +¥1,000/月

上記はあくまで「価値の置き場」の目安であり、市場での実売価格は別途検証が必要。**AIなしでも到達シナリオは組めるが、共有動画・フィードバック・トレーナー向け一覧の3点が必須レール**。

**現在地から¥20,000相当の提供価値まで、あと4回前後の大きなイテレーションが必要（AI前提のショートカットは取らない）。**

## 11. プロダクト方向性の修正（レビュー反映）

本節は v5 レポート公開後の指摘に基づき、**プロダクトの前提と優先順位を書き換える**。技術スコアの議論（§1〜§10）はそのまま活かし、**何を売るか・誰が何を見るか・ツール群をどう整理するか**だけをここで固定する。

### 11.1 AI機能について

- **採用しない**: 姿勢推定、自動フォーム採点、生成AIコーチングなどはロードマップから外す。
- **理由**: 精度・責任・コスト・プライバシーの検証コストが大きく、現フェーズのボトルネック（共有動画・トレーナー価値）を分散させる。
- **代替の価値軸**: トレーナーが**確実に**動画を見られること、メモ・コメントで介入できること、メンバーの記録が追えること。

### 11.2 「描画」と「ガイド」の差が感じられない問題

現状の実装整理（意図の違い）:

| モード | 役割（設計意図） | ユーザー体感とのギャップ |
|--------|------------------|---------------------------|
| **補助線** | プリセットのグリッド（全体に重ねる基準） | 「線が出る」という点で他と近い |
| **描画** | フリーハンドの図形（保存対象のアノテーション） | ガイドも線なので混同しやすい |
| **ガイド** | **位置可変**の水平・垂直の基準線（ドラッグで合わせる） | 描画の「直線ツール」と差が説明されていない |
| **角度** | 3点タップでの数値計測 | こちらは差別化が明確 |

**方向性**:

1. **文言・UI**: タブ名を「フリーハンド描画」「基準線（可動）」のように**機能差が一目で分かる**表記に寄せる（実装は別タスク）。
2. **中期**: どちらも「アノテーション」配下にまとめ、**保存単位・操作フローを一本化**する案を検討する（二重メンテ・学習コスト削減）。
3. **レポート上の評価**: 「角度 + 補助線」が主軸、「描画 vs ガイド」は**統合または明文化が先**でないと差別化にならない、と v5 の価値試算に反映済み（§8.3）。

### 11.3 ガイドで「水平／垂直」を押すと動画上部がぼける理由

**原因（実装ベース）**: `CustomGuides` 内のコントロールパネル（「＋水平線」「＋垂直線」ボタンを含む `div`）に `backdrop-blur-sm` が付いている。一方、このパネルは動画コンテナに対して**絶対配置ではなく**、オーバーレイ群のあとに**通常フローで配置**されるため、コンテナの**上寄り**に来やすく、背後にある `<video>` の**上端付近**が半透明＋バックドロップブラーの対象になり、**ぼけて見える**。

- 意図した「動画全体へのエフェクト」ではなく、**コントロール用パネルの CSS が動画の一部を覆っている**現象。
- **対応方針（プロダクト）**: ブラーを外す／パネルを角下固定に `absolute` する／ガイド専用のツールバーを描画タブと同じエリア（動画外の白いコントロール帯）に寄せる、のいずれかで解消を狙う。

### 11.4 ユースケースと「全会員が見える」か

**想定フロー（簡潔）**:

1. **メンバー**がアプリに登録し、ワークアウト・体組成・動画を**自分用**に記録する。
2. **トレーナー**がプロフィール上でトレーナー権限を持ち、**招待（紐付け）**で担当メンバーを増やす。
3. メンバーが記録・撮影したデータは、**RLS と `trainer_id` 関係**により、**そのメンバー本人と、紐付いたトレーナー**が参照できる範囲に閉じる（実装の詳細は `docs/architecture` 系を参照）。
4. **共有リンク**はトークンを知る者だけがメタデータ（現状は動画なし）にアクセスする別経路。

**「会員は全て見られる」想定か？ → いいえ。**  
一般メンバー同士でデータが見える設計ではない。**見えるのは「自分のデータ」＋「自分を担当に紐づけたトレーナー」＋（任意で）共有リンクの受け手**、が前提。レポート本文の「トレーナー向け SaaS」叙述と整合させる。

### 11.5 トレーナーの「ダッシュボード」は何か（現状と目標）

**現状（`/trainer`）**:

- トレーナー役割の確認。
- メンバー一覧（`trainer_id` で紐づくプロフィール）。
- メンバーごとのワークアウト件数・動画件数の**簡易カウンタ**。
- 展開時にワークアウト／動画一覧への**ナビゲーション**。

**いないもの（いわゆるダッシュボードとして不足）**:

- 週次・月次の**集約グラフ**（メンバー横断の負荷・ボリューム推移）。
- 「要フォロー」などの**優先度付きキュー**。
- 新着（動画アップロード・ワークアウト記録）の**ダイジェスト**。
- ダッシュボードというより **「メンバー管理＋深掘りへの入口」** に近い。

**方向性**: 「ダッシュボード」という語を安易に使わず、まず **(A) 共有動画でレビューできる** → **(B) コメント等のフィードバック** → **(C) トレーナー向けの進捗ビュー（一覧＋推移）** の順で厚みを足す。§9 の N-P1-1 / N-P1-2 と同一優先度で追記する。
