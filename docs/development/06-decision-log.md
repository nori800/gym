# Decision Log（ADR）

実装でブレが生じやすい設計判断を記録する。変更時は旧決定を ~~取り消し線~~ で残し、日付と理由を追記する。

**関連**: [requirements/08-user-flow-and-steps.md](../requirements/08-user-flow-and-steps.md)、[architecture/05-screen-transitions.md](../architecture/05-screen-transitions.md)

## ADR-001: 未ログイン時の撮影可否

| 項目 | 内容 |
|------|------|
| ステータス | **決定済み（MVP）** |
| 日付 | 2026-04-20 |
| 決定 | **MVP ではアプリ起動後、必ずログイン／新規登録を通す。** ただし UX 上のメイン導線は「撮影へ」を最短で提示する（ホームの CTA）。 |
| 理由 | ローカル保存→後からアップロードの実装コストが高い（IndexedDB容量制限、同期衝突）。MVP はシンプルに認証後すべてクラウド保存とし、ステップ番号は概念上の優先度として維持する。 |
| 代替案 | ゲスト撮影→ログイン後アップ（Phase2 以降で検討可能） |

## ADR-002: 種目マスタの形式

| 項目 | 内容 |
|------|------|
| ステータス | **決定済み（MVP）** |
| 日付 | 2026-04-20 |
| 決定 | MVP では **固定リスト（シードデータ）＋ユーザーによるカスタム追加**。カスタム種目は `profiles.favorite_exercises` に JSON 配列で保持する。 |
| 理由 | 種目マスタテーブルを別途つくると Phase1 でマイグレーション管理が増える。ユーザーが少ないうちは JSON で十分。Phase2 でタグ正規化テーブルへ移行可能な設計にしておく。 |
| 代替案 | 完全フリー入力のみ / 正規化テーブル |

## ADR-003: 動画のアップロード上限（MVP初期値）

| 項目 | 内容 |
|------|------|
| ステータス | **仮決定（検証後に調整）** |
| 日付 | 2026-04-20 |
| 決定 | 1ファイル最大 **100 MB**、長さ最大 **5分**。クライアント側で事前チェックし、超過時はメッセージ表示。 |
| 理由 | Supabase Free プランの Storage 帯域を考慮。一般的なスマホ筋トレ動画（1080p 30fps）は 1分 ≈ 15–30MB に収まることが多い。 |
| 代替案 | サーバー側トランスコード、分割アップロード |

## ADR-004: 状態管理ライブラリ

| 項目 | 内容 |
|------|------|
| ステータス | **決定済み（MVP）** |
| 日付 | 2026-04-20 |
| 決定 | **React のビルトイン（useState / useContext）+ Server Components のデータフェッチ** を基本とする。クライアント状態が複雑化した場合に限り、**Zustand** を最小限で導入する。Redux は使わない。 |
| 理由 | Next.js App Router は Server Components 主体で、グローバル状態が少ない。Zustand は軽量で学習コストが低い。 |

## ADR-005: パッケージマネージャ

| 項目 | 内容 |
|------|------|
| ステータス | **決定済み（変更あり）** |
| 日付 | 2026-04-20 |
| 決定 | ~~pnpm を使う。~~ → **npm** を使う。環境の権限制約により corepack/pnpm のグローバル導入が困難なため。`package-lock.json` をコミット対象に含める。 |
| 理由 | Node 同梱の npm であれば追加インストール不要で確実に動く。 |

## ADR-006: 実装順序（UI 先行・Supabase は後段）

| 項目 | 内容 |
|------|------|
| ステータス | **決定済み** |
| 日付 | 2026-04-20 |
| 決定 | **まず UI/UX・画面遷移・モックデータでアプリ全体を動かし、Supabase（Auth・DB・Storage）の連携は後続フェーズで行う。** 履歴・一覧・記録は開発途中 **ダミーデータでよい**。 |
| 理由 | 見た目と操作感の確定を優先し、バックエンド仕様の細部と並行すると UI がブレるのを防ぐ。データ層は [architecture/06-supabase-physical-design.md](../architecture/06-supabase-physical-design.md) で固定済みのため、後から接続可能。 |
| 補足 | **完全な MVP 受け入れ**（ログイン・永続化・RLS）は Supabase フェーズ完了後に満たす。UI 先行フェーズでは認証画面を **モック**または **スキップ可能**としてよい（[ADR-001](06-decision-log.md) は本番接続時に適用）。 |
| 参照 | [09-implementation-plan.md](09-implementation-plan.md) |
